# 简历策略参考手册

> 本文档为 resume-builder skill 的核心策略参考文件，基于中国互联网行业校招的真实规律整理。
> 适用于计算机/软件工程相关专业本科及以上学历，目标岗位为后端开发、前端开发、算法工程师、
> 数据科学、测试开发、客户端开发、运维/SRE 等互联网技术岗。

---

## 一、简历策略总则

### 1.1 核心竞争力公式

简历的最终竞争力由以下四个维度共同决定，不同学历层次需要不同权重分配来最大化
面试转化率：

```
简历竞争力 = 学历背书 × 项目实力 × 实习经验 × 技术深度
```

这四个维度并非简单相加，而是存在交互效应：

- **学历背书**是简历筛选阶段的"敲门砖"，影响简历能否被 HR 看到
- **项目实力**是面试阶段的核心战场，直接决定能否通过技术面
- **实习经验**是学历和项目之间的"桥梁"，一段大厂实习可以部分弥补学历不足
- **技术深度**贯穿始终，决定了在任何一个维度上你能走多远

### 1.2 策略矩阵

不同学历层次的最优策略完全不同。一刀切的简历写法会导致竞争力大幅折损。

| 学历层次 | 教育背景权重 | 项目经历权重 | 实习经历权重 | 技术栈策略 | 推荐项目数 | 板块顺序 |
|---------|------------|------------|------------|-----------|-----------|---------|
| 985/双一流 | ★★★ (前置) | ★★★ | ★★ | 1-2 方向深挖 | 2-3 | 教育→实习→项目→技能 |
| 211 | ★★ (前置) | ★★★ | ★★★ | 1-2 方向深挖 | 3-4 | 教育→实习→项目→技能 |
| 双非一本 | ★ (后置) | ★★★ | ★★★ | 1 方向精通 | 3-4 | 技能→项目→实习→教育 |
| 二本/专科 | ☆ (后置/弱化) | ★★★ | ★★★ | 1 方向精通+广度 | 4-5 | 技能→项目→实习→教育 |

**关键原则：**

- 985/211：利用学校背书获得初筛通过，用深度项目证明自己不只是"会考试的"
- 双非/二本：教育背景不可改变，必须在项目和实习上做到"一眼就能看出强"
- 板块顺序不是固定的——双非学生把教育背景放第一屏是在主动给自己挖坑

### 1.3 各维度的深层含义

**教育背景不只是学校名字：**
- 985/211：写 GPA（如果 >3.5/4.0 或专业前 30%）、相关课程（仅写高分/硬核课程）、排名
- 双非：如果 GPA 高（专业前 10%）可以写，否则只写学校、专业、时间
- 二本/专科：GPA 高则写排名不写绝对值（"专业前 5%"比"GPA 3.8"更有说服力）

**项目经历不只是代码仓库：**
- 面试官判断项目质量的标准：有没有真实用户？有没有线上 demo？有没有持续维护？
- "课程设计"和"真实项目"在面试官眼里的差距是天壤之别
- 没有线上可访问 URL 的项目 = 没有被验证的项目

**实习经历不只是公司名字：**
- 一段平庸描述的字节实习 < 一段量化结果突出的中小厂实习
- 实习描述必须回答：你做了什么？用了什么技术？产生了什么影响？
- 没有量化结果的实习 = 没有说服力的实习

## 二、各学历层次的具体策略

### 2.1 985/双一流策略

**核心思路：** 学历背书帮你拿到面试，项目深度帮你通过面试。

**教育背景处理：**
- 位置：简历前 1/4 处，紧接个人信息之后
- 内容：学校全称 + 专业 + 学位 + 时间 + GPA（高则写）+ 排名（好则写）
- 可选加分项：国奖/校奖、优秀毕业论文、辅修/双学位
- 不要写：无关通识课成绩、高中获奖

**项目策略：**
- 重质量不重数量，2-3 个深度项目足够
- 每个项目至少要有：技术选型的理由、踩过的坑、性能优化的量化结果
- 实验室/导师的科研项目非常适合包装——但要强调你的个人贡献而非项目本身
- 警惕"985 学生常见的工程能力不足"印象：确保至少一个项目体现工程落地能力

**实习策略：**
- 大厂实习优先于多段小厂实习（一段字节/阿里 > 三段不知名公司）
- 如果没有大厂实习：用科研项目 + 开源贡献补位
- 实验室与企业合作的横向项目可以当作实习来写（但要说清楚你在其中的角色）

**特别注意：**
- 不要过度依赖学校名——面试官见到 985 简历反而会提高期望值
- 项目描述敷衍是面试挂掉的头号原因：每个技术名词都要经得起追问
- "我只是负责写 CRUD"这种回答比没做过项目更致命

### 2.2 211 策略

**核心思路：** 在 985 和双非之间——学历有优势但不足以躺平，项目和实习需要并重。

**教育背景处理：**
- 位置：简历前 1/3 处
- 要点：突出专业排名（如果是优势专业）、相关课程、学术成果
- 211 的优势专业（如北邮的通信、西电的计算机）可以在学校旁边标注专业排名

**项目策略：**
- 3-4 个项目，至少 1 个有线上 demo
- 建议项目构成：1 个基础项目 + 1 个进阶项目 + 1 个有真实用户的项目 + 1 个开源相关
- 技术栈要体现聚焦：选定一个主方向（后端/前端/算法），不要多方向并行

**实习策略：**
- 实习对 211 学生的重要性高于 985——因为学历筛选中 211 处于"可过可不过"区间
- 一段大厂实习可以让你在简历筛选中和 985 站在同一起跑线
- 实习路径：大二暑假中型公司 → 大三寒假大厂日常实习 → 大三暑假大厂暑期实习

**特别注意：**
- 211 学生的简历容易"高不成低不就"：既没有 985 的学历光环，又没有双非的破釜沉舟
- 核心解决思路：让你的项目和实习有明确的"记忆点"

### 2.3 双非一本策略（重点）

**核心思路：** 学历是劣势，但可以通过项目质量和实习经验完全翻盘。

#### 2.3.1 教育背景处理

**位置：必须后置。** 把教育背景放在简历后半部分（技术栈和项目经历之后），
让 HR 先看到你的技术实力再去查学历。这不是"隐藏"，而是"优先级排序"。

- 写法："XX大学 计算机科学与技术 本科 20XX-20XX"
- 如果 GPA 高（专业前 10%），写排名不写绝对 GPA 值
- 如果 GPA 一般，只写学校、专业、时间，不写 GPA
- 不要写无关课程（大学英语、思修、体育）

#### 2.3.2 项目策略——双非简历的核心战场

**必须用 3-4 个高质量项目撑起简历。** 项目是双非学生最重要的竞争力来源。

**项目质量标准（按优先级排序）：**
1. 有真实用户在使用的项目（哪怕只有几百个用户）
2. GitHub 有 star（100+ 就算亮点）
3. 有完整的文档（README + Wiki + API 文档）
4. 有对应的技术博客（记录踩坑和思考）
5. 有线上 demo 可以立即访问
6. 不是课程设计（图书馆管理系统、学生成绩管理系统 = 直接扣分）

**项目构成建议：**
- 项目 1（明星项目）：独立维护、有用户、有 star、有文档——这是你的"代表作"
- 项目 2（深度项目）：体现某个方向的深度，如自研 RPC 框架、手写简单数据库
- 项目 3（广度项目）：体现全栈能力或跨领域能力
- 项目 4（可选）：开源贡献项目 or 竞赛相关项目

**技术栈要求：**
- **只写一个方向精通。** 不要写"精通 Java + Python + C++ + Go + JavaScript"
  - 这种写法在双非简历里 = 告诉面试官"我一个都不精"
  - 正确写法：精通 Java 生态（Spring Boot/MyBatis/MySQL），熟悉 Python 用于数据处理
- 技术栈要具体到框架和工具，不要只写语言名
  - ❌ "熟悉 Java"
  - ✅ "熟悉 Java，掌握 Spring Boot 2.x/Spring Cloud/JVM 调优"

#### 2.3.3 实习策略——双非翻盘的关键

**实习比项目更重要。** 有一段大厂实习就能大幅提升简历通过率。

**真实路径规划：**

- **路径 1：层层递进（最安全）**
  - 大二暑假：小公司/创业公司实习（代码能力 + 有东西可写）
  - 大三寒假：中型互联网公司实习（积累经验 + 刷简历）
  - 大三暑假：目标大厂实习（终极目标）

- **路径 2：开源突围（适合编程能力强但没实习的人）**
  - 大二开始持续给知名项目提 PR
  - 维护自己的开源项目并做到 100+ star
  - 参与 GSoC/OSPP 等开源实习项目
  - 简历写"XX 项目核心贡献者"比写"XX 小厂实习生"更有力

- **路径 3：竞赛突围（适合算法方向）**
  - Kaggle 银牌/金牌 → 数据科学/算法岗敲门砖
  - 天池/Datawhale 比赛 Top 10% → 简历亮点
  - ACM 区域赛铜牌以上 → 代码能力的有力证明
  - 蓝桥杯国奖 → 至少证明了基础编程能力

**小厂实习的写法要领：**
- 即便公司不有名，也要把做的事情写清楚、写量化
- ❌ "负责公司内部管理系统开发"
- ✅ "独立开发公司内部 CRM 系统，重构旧系统后将查询响应时间从 3s 降至 200ms，支撑 50+ 销售日常使用"

#### 2.3.4 双非生存法则（面试准备）

1. **准备好"为什么选这个技术方案"的回答。**
   - 面试官对双非学生的技术方案会更质疑，你必须能说清楚每个选择
   - "因为网上教程这么写的" = 直接挂

2. **准备好证明你代码能力的证据链：**
   - GitHub 活跃记录（绿点墙）
   - 技术博客 10+ 篇
   - 开源项目的 merged PR
   - LeetCode 刷题记录（300+ 题）

3. **准备好"你的优势是什么"的回答：**
   - 不要回避学历问题，主动转化为优势：
   - "虽然我在 XX 双非学校，但这让我更早意识到需要靠实际项目来证明自己……"
   - 把话题引导到你的项目和实习上

### 2.4 二本/专科策略

**核心思路：** 学历是硬伤，需要指数级超过同岗位其他候选人的项目/实习来弥补。

**教育背景处理：**
- 位置：简历最后，甚至只写学校、专业，不写时间（避免被直接年龄推算）
- 写法：极简——"XX学院 计算机科学与技术 本科"（10 个字以内解决）
- 如果 GPA 非常高（专业第一），可以写排名，否则不写
- 如果是专科升本，建议只写本科学校（面试时诚实说明即可）

**项目策略：**
- **4-5 个项目是底线。** 你必须比别人多 1-2 个项目来证明自己
- **必须有一个"明星项目"：**
  - 标准：独立维护 + 有真实用户或开源有 star + 有完整文档 + 有技术博客
  - 这个项目是你面试时的主战场，面试官 80% 的问题会围绕它
- **技术栈要求：一个方向精通 + 有全栈广度**
  - 后端方向：要能独立完成从前端页面到数据库的全链路
  - 前端方向：要能独立搭建 CI/CD + 服务端渲染 + 性能优化
  - 全栈能力对二本/专科学生是加分项，不是减分项——因为小公司更需要全栈

**实习策略：**
- 实习路径：小厂 → 中厂 → 大厂外包/子公司 → 大厂正编
- 大厂外包不是羞耻——可以在外包岗位证明自己后转为正编
- 内推是二本/专科学生最重要的渠道——海投的简历大概率被直接筛掉
- 考虑：大厂的区域性分公司/研发中心（如武汉、成都、西安）竞争相对小

**渠道策略：**
- 不要只盯着官网投递——那是为 985/211 准备的
- 有效渠道：技术社区内推、校友推荐、猎头、招聘会现场投递
- GitHub 上的招聘帖（如"XX 团队直招"）往往对学历要求更宽松

## 三、大厂简历筛选标准

### 3.1 简历筛选 6 秒法则

HR 平均只用 6 秒扫一份简历来决定是否进入下一轮。这意味着：

- **简历的第一屏（前 1/2）决定了命运。**
- 关键词必须能被快速扫描到：技术栈、公司名、学校名、量化数字
- 大段文字 = 没人看 = 被跳过

**6 秒内的视线动线（眼动实验结论）：**
1. 秒 1-2：扫个人信息和求职意向
2. 秒 2-3：扫第一段经历（实习 or 项目，取决于谁在最前面）
3. 秒 3-4：扫技术栈关键词
4. 秒 4-6：扫数字和亮点，或直接跳过

**结论：前 2 秒没看到亮点 = 被淘汰。**

### 3.2 关键词匹配机制

大部分大厂使用 ATS（Applicant Tracking System）做初筛，简历会被扫描以下关键词：

**必须命中的关键词类型：**
- 技术栈：Java/Python/Go/C++/JavaScript/React/Vue/Spring/MySQL/Redis 等
- 学历关键词：本科/硕士/985/211（仅用于分类，不是自动淘汰）
- 经验关键词：实习/项目/开源/竞赛
- 岗位关键词：后端/前端/算法/数据/测试

**关键词优化技巧：**
- 技术栈要同时写缩写和全称，如 "MySQL（关系型数据库）"
- JD 里出现的所有技术栈，只要你会，都要出现在简历里
- 但不要堆砌——写到技能板块即可，不要项目描述里乱塞

### 3.3 量化结果的强制要求

**每个项目/实习描述必须至少有一个数字。** 这是大厂面试官筛选简历的最基本标准。

**量化结果示例（按类型）：**

| 类型 | 不好的写法 | 好的写法 |
|------|----------|---------|
| 性能优化 | 提升了系统性能 | QPS 从 500 提升至 3000（6 倍） |
| 用户量 | 有很多用户使用 | 上线 3 个月积累 2000+ 注册用户 |
| 代码质量 | 减少了代码 bug | 单元测试覆盖率从 30% 提升至 85% |
| 效率提升 | 提高了开发效率 | 构建时间从 5 分钟缩短至 30 秒 |
| 数据量 | 处理了大量数据 | 日处理日志 50GB+，数据总量 10TB+ |
| 开源影响 | 项目有人关注 | GitHub 500+ star，被 XX 组织收录 |
| 业务影响 | 负责重要功能开发 | 负责核心支付模块，日均交易额 100W+ |

**量化结果的四个来源：**
1. 实际测量的数据（基准测试、压测结果）
2. 用户统计数据（GitHub Insights、友盟、Google Analytics）
3. 估算数据（"日均处理请求约 XX 万次"）
4. 对比数据（"相比旧方案，响应时间降低 60%"）

**注意：** 数字必须真实可验证。面试官可能会追问"你这个数字是怎么算出来的"。
如果回答不上来，比没有数字更糟糕。

### 3.4 STAR 完整性检查

每个项目描述应当遵循 STAR 原则，缺 Result 的项目会被认为没有完成。

- **S (Situation):** 背景和问题是什么？（1 句话，可选）
- **T (Task):** 你要完成什么任务？（1 句话）
- **A (Action):** 你做了什么？（3-5 个 bullet points）
- **R (Result):** 取得了什么成果？（1-2 个量化结果）

**各部分的占比建议：**
- Action 占 60-70% 的篇幅（面试官最关心的部分）
- Result 占 20-30%（必须有数字）
- Situation + Task 占 10%（1 行背景交代即可）

### 3.5 一页原则

**应届生简历超过一页直接扣分。** 这是所有大厂 HR 的共识，没有例外。

**一页简历的内容密度上限（A4 纸）：**
- 个人信息 + 求职意向：3-4 行
- 教育背景：3-5 行
- 实习经历：10-15 行（2-3 段）
- 项目经历：15-20 行（3-4 个）
- 专业技能：5-8 行
- 其他（竞赛/证书/博客）：3-5 行

**控制篇幅的技巧：**
- 删掉所有"熟练掌握 Office 办公软件"类无意义条目
- 删掉和求职方向无关的经历（如服务员兼职、学生会活动）
- 技术栈用紧凑排版（两栏并列，不用一栏一行）
- 项目描述控制在 3-4 个 bullet points，每个 1-2 行
- 页边距缩小到 1.5cm 左右（但不要小于 1cm）

### 3.6 格式红线

以下任何一条都会导致简历被直接淘汰或严重扣分：

- **❌ 提交 Word 格式简历。** 必须用 PDF。Word 在不同设备上排版会乱。
- **❌ 有错别字或语法错误。** "熟练使用"写成"熟练实用" → 直接挂。
- **❌ 使用花哨模板。** 多彩色、多图标的模板不适合互联网行业。
- **❌ 英文简历有明显语法错误。** 如果有英文版，找英语好的同学检查。
- **❌ 电话号码或邮箱写错。** 每年都有因为联系方式错误联系不上的候选人。
- **❌ 照片使用生活照或自拍。** 要么不放照片，要么放正式证件照。
- **❌ 文件命名随意。** 应该用"姓名_学校_岗位_简历.pdf"格式。

### 3.7 双非简历的特殊生存法则

**"可视区"法则：**
- 双非简历：项目和实习必须出现在简历前 1/3 的可视区域
- 技术博客 URL 和 GitHub 地址要放在个人信息旁边，显眼但不突兀
- 教育背景必须在翻页后才能看到（即不在第一屏）

**"关键词密度"法则：**
- 双非简历的技术关键词密度应该比 985 简历高 30% 以上
- 原因：HR 扫描时会更仔细地看双非简历（出于确认偏误），更高密度的关键词能"说服"HR

**"信任背书"法则：**
- 双非学生需要额外的信任背书来弥补学历信任缺失
- 有效背书（重要性递减）：大厂实习 > 知名开源项目贡献 > 竞赛获奖 > 技术博客 > GitHub star
- 需要至少 2 种不同类型的背书

**严格禁止的条目：**
- "熟练使用 Office 办公软件" —— 这是技术岗，不是行政岗
- "通过 CET-4" —— 除非外企，否则不需要写英语成绩
- "学习能力强，有团队精神" —— 这是最廉价的自我评价，删掉
- "熟练使用 Windows/Linux 操作系统" —— 这意味着你真的没有东西可写
- 各种在线学习平台的证书（Coursera/Udemy）—— 放在 LinkedIn 上，不要放简历上

## 四、真实成功案例模式

以下案例模式基于真实求职经验总结，不是虚构案例，但做了匿名化处理。

### 4.1 案例模式一：项目驱动型（适合双非 / 没有实习的学生）

**背景特征：**
- 学历：双非一本或以下
- 实习：无或仅有一段小公司实习
- 项目：3-4 个高质量项目
- 最终去向：大厂后端/前端开发岗

**简历特征：**
- 教育背景放最后
- 第一屏是"专业技能 + 明星项目"
- 技术栈紧凑排列（两栏，而非一栏一行）
- 4 个项目，每个 4-5 个 bullet points

**项目构成：**
1. **明星项目（简历核心）：** 一个独立开发的、有线上 demo 的全栈项目。如：
   - 一个技术博客系统（但绝不是简单的 CRUD——包含 Markdown 渲染、全文搜索、访问统计、SEO 优化）
   - 一个即时通讯应用（WebSocket + 消息队列 + 已读未读状态管理）
   - 一个 API 网关/统一认证平台
2. **轮子项目（体现深度）：** 手写 RPC 框架 / 简易 ORM / 简易消息队列 / 简易数据库
3. **协作项目（体现团队能力）：** 开源项目贡献 or 多人协作的 side project
4. **课程项目转型（备选）：** 把课程设计升级改造，加入生产级特性（监控、日志、CI/CD）

**关键成功因素：**
- 每个项目都有 GitHub 仓库 + 完善的 README（项目背景、技术架构图、快速开始指南）
- 至少一个项目有线上可访问的 URL
- 技术博客 10+ 篇，记录项目开发过程和踩坑经验
- GitHub profile 持续活跃（绿点墙不是空白的）
- 准备了每个项目的"技术选型答辩"（为什么用这个框架/数据库/架构）

**面试表现：**
- 面试官会花 60% 以上的时间问明星项目
- 如果明星项目质量过硬，学历问题会被自然淡化
- 关键是要能说出来"为什么要这样设计"和"遇到过什么困难"

### 4.2 案例模式二：实习积累型（适合各类院校）

**背景特征：**
- 学历：211 及以上更有利，双非也可走通
- 实习：2-3 段，层层递进
- 项目：2-3 个（可以包含实习做的项目）
- 最终去向：大厂核心部门

**实习时间线：**
- 大二暑假（7-9 月）：第一段实习——小公司/创业公司
  - 目标：获得真实工作经历，有东西可写
  - 岗位不重要，公司大小不重要，重要的是你做了什么
- 大三寒假（1-3 月）：第二段实习——中型公司/大厂日常实习
  - 目标：积累规模化的工程经验
  - 最好和最终目标岗位方向一致
- 大三暑假（7-9 月）：第三段实习——目标公司暑期实习
  - 目标：拿到 return offer
  - 暑期实习的 return 率通常在 50-70%

**简历特征：**
- 实习经历放最前面（教育背景根据学历决定前置或后置）
- 每段实习写 3-4 个 bullet points，重点写自己的贡献
- 实习做过的项目可以单独列在项目经历里（但要标注"实习期间"）
- 从第一段到第三段，描述的复杂度递增（体现成长曲线）

**关键成功因素：**
- 每段实习都有量化结果（"负责 XX 模块开发，支撑 XX 万用户"）
- 实习经历之间能看到技能和复杂度的递进
- 较早开始规划——大二就开始行动
- 实习期间主动要活、主动学习、主动总结（这些都是面试时的素材）

### 4.3 案例模式三：竞赛突围型（适合算法/ML/数据科学方向）

**背景特征：**
- 学历：不限（竞赛成绩可以部分弥补学历）
- 方向：算法工程师、数据科学家、机器学习工程师
- 竞赛：有含金量的比赛获奖
- 最终去向：大厂算法岗

**有效竞赛清单（按含金量排序）：**
1. **Kaggle：** Solo 金牌/银牌/铜牌 > Team 金牌 > Team 银牌
2. **天池/Datawhale：** Top 10 或 Top 5%
3. **ACM-ICPC：** 区域赛金牌/银牌/铜牌
4. **蓝桥杯：** 国赛一等奖/二等奖（省赛的含金量低很多）
5. **数学建模（国赛/美赛）：** 国赛一等奖+/美赛 M 奖+（仅对算法岗有一定参考）
6. **CCF 相关比赛：** CSP 高分、CCSP 获奖

**简历特征：**
- 竞赛成绩放教育背景下方或个人信息旁边（显眼但不突兀）
- 项目经历可以包含竞赛项目（但要包装成工程化描述）
- 如果有论文（哪怕不是一作），附在竞赛后面

**论文的处理：**
- 有论文是加分项，但仅对算法岗有效（对后端/前端影响不大）
- 写论文引用格式，标注自己是不是一作
- 即便是二作/三作也可以写——说明你有学术经历

**关键成功因素：**
- 竞赛成绩要"好到能让人忽略学历"
- 除了竞赛，还需要至少一个工程化项目（纯竞赛选手容易被认为"只会打比赛"）
- 竞赛方案要能写成技术博客（证明你不只是刷了比赛）

### 4.4 案例模式四：开源贡献型

**背景特征：**
- 学历：不限
- 特点：在开源社区有显著贡献
- 最终去向：大厂（经常被直接内推或联系）

**有效的开源贡献（按含金量）：**
1. 知名项目（Apache/Django/Spring/Vue.js 级别）的 merged PR——最高含金量
2. 知名项目的重要 issue 修复或功能实现
3. 自己维护的项目有 100+ GitHub star
4. 自己维护的项目被其他项目依赖
5. 参与 GSoC (Google Summer of Code) 或 OSPP (开源之夏)
6. 知名技术社区（如掘金/SegmentFault）的活跃作者

**简历特征：**
- GitHub profile URL 放在个人信息旁边
- 开源项目的链接直接写在项目经历里
- "XX 项目核心贡献者"比"参与了 XX 项目"更有力

**关键成功因素：**
- 贡献要持续而非一次性（持续 6 个月以上的贡献才被认可）
- 不只是修 typo，要有实质性的代码贡献
- 开源项目的维护者可能成为你的推荐人

## 五、常见简历错误（按学历分层）

### 5.1 双非学生常见错误

**错误 1：教育背景放最前面**
- 问题：HR 打开简历第一眼看到双非学校 → 可能直接关掉
- 正确做法：先用项目和技术实力抓住眼球，教育放最后

**错误 2：技术栈写太多太杂**
- 问题："精通 Java、Python、C++、Go、JavaScript、Rust……" 超过 10 个
- HR/面试官心理：双非说精通这么多 → 大概率是一个都没用过几次
- 正确做法：1 个方向写"精通/熟练"，1-2 个方向写"了解/熟悉"

**错误 3：项目全是课程设计**
- 问题：图书馆管理系统、学生成绩管理系统、网上商城——面试官见了想吐
- 正确做法：要么做有真实场景的项目，要么把课程设计改造成"不像课程设计"
- 改造方法：加上 Docker 部署、CI/CD、监控、压测、API 文档、Swagger

**错误 4：没有量化结果**
- 问题："提升了系统性能""提高了开发效率"——没有数字等于没说
- 正确做法：每个项目至少写 1-2 个数字

**错误 5：没有任何线上可访问的项目**
- 问题：GitHub 上只有代码，没有 README，没有 demo 链接
- 面试官不会 clone 你的代码来跑——他们只看 README 和 demo
- 正确做法：项目必须有 README + 架构图 + 快速开始 + demo URL（或截图/GIF）

**错误 6：求职意向不明确**
- 问题：同时写"后端开发/前端开发/算法工程师"
- 这等于告诉 HR"我自己都不知道想做什么"
- 正确做法：一个方向一份简历，不要通用版

**错误 7：自我评价空洞无物**
- 问题："学习能力强，有团队合作精神，能承受工作压力"
- 正确做法：要么不写自我评价，要么用事实代替形容词
  - ✅ "主导 XX 项目从 0 到 1 的开发，协调 3 人团队完成交付"

**错误 8：把"了解"写成"精通"**
- 双非学生写"精通"的门槛要比 985 更高——面试官会更较真
- 如果只在 tutorial 里用过，请写"了解"或"熟悉"
- 用过的标准：能在生产环境中独立使用并解决问题

### 5.2 985/211 学生常见错误

**错误 1：过度依赖学校名**
- 问题：认为"我是 985 的嘛，简历随便写写就行"
- 现实：大厂面试官见过的 985 简历太多了，学校名只是让你进入筛选，不是通过面试
- 正确做法：学校名是敲门砖，项目质量是通行证——两者缺一不可

**错误 2：项目描述敷衍**
- 问题：项目描述只有 1-2 行，没有技术细节，没有量化结果
- 面试官预期：985 学生应该有更强的技术表达能力
- 敷衍的项目描述 = "这个学生可能只是考试成绩好"

**错误 3：技术栈太学术化**
- 问题：只写论文用过的工具（MATLAB、LaTeX、某学术框架），不体现工程能力
- 大厂要的是工程能力，不是论文发表能力（除非投算法岗）
- 正确做法：学术项目也要包装成工程化描述

**错误 4：没有实习**
- 问题：认为学校好就不需要实习——这是最大的误区
- 大厂校招中，有实习的 985 > 没实习的 985
- 正确做法：大二/大三必须至少有一段实习（哪怕不是大厂）

**错误 5：自我评价过于谦虚**
- 985 学生反而容易写得太谦虚："基本掌握了 XX""对 XX 有一定了解"
- 实际上你的"基本掌握"可能已经是双非学生的"精通"水平
- 正确做法：客观描述你的能力，不要过度谦虚

**错误 6：简历超过一页**
- 问题：想把所有获奖、课程项目、社团活动都写上去
- 结果：简历 2-3 页 → HR 直接略过
- 正确做法：只写和求职方向相关的，删掉无关内容

### 5.3 所有学生都容易犯的错误

**通用错误 1：一份简历投所有岗位**
- 每个岗位方向需要定制化简历
- 后端岗突出并发/数据库/系统设计
- 前端岗突出 UI 组件/性能优化/浏览器原理
- 算法岗突出数学基础/论文/竞赛

**通用错误 2：没有为 AI 初筛优化**
- 越来越多公司用 AI 做初筛
- AI 筛简历看：关键词匹配度、量化结果密度、格式规范性
- 优化方向：增加技术关键词密度、确保每段经历有数字、使用标准格式

**通用错误 3：联系方式不明显**
- 电话号码和邮箱应该放在最显眼的位置（通常在名字下方）
- 有些人把联系方式藏在简历底部——HR 找不到 = 联系不上 = 淘汰

**通用错误 4：写期望薪资（应届生）**
- 应届生校招薪资是固定的（白菜/sp/ssp），不需要也不应该写期望薪资
- 写了反而可能压低 package 或直接被过滤

**通用错误 5：使用个人照片作为简历亮点**
- 国内互联网行业不放照片是常态，放了不会加分
- 放自拍/生活照反而会扣分（显得不专业）

## 六、如何在 Skill 的 Phase 2 中应用本文档

### 6.1 策略加载流程

```
Phase 1: 信息收集 → 确认用户学历
    ↓
加载对应策略（从本文档读取）
    ↓
Phase 2: 简历编排 → 应用策略到简历结构和内容
```

### 6.2 学历确认后的自动策略应用

当 Phase 1 确认用户的学历层次后，skill 应自动加载以下策略并应用到 Phase 2：

**1. 板块顺序自动调整：**
- 985/211：个人信息 → 教育背景 → 实习经历 → 项目经历 → 专业技能
- 双非/二本/专科：个人信息 → 专业技能 → 项目经历 → 实习经历 → 教育背景

**2. 推荐项目数量自动调整：**
- 985/双一流：推荐 2-3 个项目，强调质量
- 211：推荐 3-4 个项目
- 双非一本：推荐 3-4 个项目，必须有 1 个明星项目
- 二本/专科：推荐 4-5 个项目，必须有 1 个明星项目

**3. Phase 2 开始前的策略提示：**

Skill 应根据用户学历，在进入 Phase 2（简历编排）之前，展示对应的策略要点摘要。

示例（双非用户）：
```
📋 简历策略提示（基于你的学历背景）：

1. 板块顺序：技术栈和项目放在前面，教育背景放在最后
2. 项目数量：建议 3-4 个项目，至少有 1 个"明星项目"
3. 技术栈：聚焦 1 个方向，不要写太多"精通"
4. 量化要求：每个项目至少 2 个量化结果
5. 额外要求：项目需要有线上 demo 或 GitHub 完善文档

需要我详细解释任何一条策略吗？还是我们直接开始写简历？
```

**4. 审查阶段的针对性检查：**

Phase 5（审查与输出）时，根据学历针对性地检查常见错误：

- 双非用户：检查教育背景是否后置、技术栈是否过于宽泛、项目是否有量化结果
- 985 用户：检查项目描述是否敷衍、是否有实习经历、是否有量化结果

### 6.3 用户画像标记

Skill 应在内部维护一个用户画像标记（不输出给用户），用于在各 Phase 中做决策：

```yaml
user_profile:
  education_tier: "双非一本"  # 985/211/双非一本/二本/专科
  education_position: "back"  # front/back
  recommended_projects: 4     # 2-5
  focus_area: "后端"           # 后端/前端/算法/...
  strategy_patterns:           # 推荐的案例模式
    - "项目驱动型"
    - "实习积累型"
  critical_checks:             # 审查时必须检查的错误
    - "education_position_correct"
    - "tech_stack_not_too_broad"
    - "projects_have_quantitative_results"
    - "has_star_project"
```

### 6.4 与其他参考文件的协作

- **resume-template.md：** 提供简历的 LaTeX 模板，本文件决定模板的使用方式
- **project-templates.md：** 提供项目模板，本文件决定推荐哪些类型/数量的项目
- **career-directions.md：** 提供方向选择参考，本文件补充"选定方向后如何写"
- **interview-prep.md：** 提供面试准备策略，本文件的 STAR 原则和量化要求是面试的基础

### 6.5 动态调整规则

以下情况应触发策略调整：

| 触发条件 | 策略调整 |
|---------|---------|
| 用户有 1 段以上大厂实习 | 学历权重可降低，实习权重提升 |
| 用户 GitHub 有 500+ star 的项目 | 项目权重可提升，可减少推荐项目数量 |
| 用户有 Kaggle 金银牌或 ACM 奖牌 | 加入竞赛突围策略，学历权重降低 |
| 用户专业非计算机但目标技术岗 | 需要更强的项目质量弥补专业偏差 |
| 用户有论文发表（一作/EI/SCI） | 算法岗: 论文前置；工程岗: 论文放最后 |
| 用户目标是外企或远程岗位 | 附加英文简历策略，英语能力展示要求 |
