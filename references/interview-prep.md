# 面试准备模块 (Phase 5) — 参考文档

> 本文档定义简历技能工具 Phase 5（面试准备）的完整流程与规则。
> 在用户完成简历构建（Phase 1-2）与项目实现（Phase 3-4）后，进入面试准备阶段。

---

## 一、从简历内容自动生成面试问题

### 1.1 总体规则

- 为简历中每个"可考点"生成 3-5 个问题
- 难度递进：基础概念 → 实战应用 → 深层原理
- 每题标注对应简历字段来源，便于追溯
- 所有问题以中文呈现，技术术语保留英文原名

### 1.2 从「技术栈」生成问题

对简历中列出的每一项技术，按如下模板生成问题：

```
【难度递进模型】
  Q1 (基础): 核心概念、常用 API、基本用法
  Q2 (实践): 实际项目中的使用方式、常见坑、最佳实践
  Q3 (原理): 底层实现、源码级理解、设计权衡
```

**示例 — Redis：**

| 编号 | 问题 | 难度 | 类别 |
|------|------|------|------|
| 1 | Redis 支持哪些数据结构？各自适用什么场景？ | 基础 | 概念 |
| 2 | Redis 如何实现分布式锁？用 SETNX 有什么问题，Redlock 又解决了什么？ | 实践 | 应用 |
| 3 | Redis 持久化 RDB 和 AOF 的区别是什么？各自的实现原理和优缺点？ | 原理 | 底层 |
| 4 | Redis 集群模式有哪几种？数据分片（slot）是如何工作的？ | 实践 | 架构 |
| 5 | Redis 内存淘汰策略有哪些？LRU 和 LFU 在 Redis 中是如何近似实现的？ | 原理 | 底层 |

**示例 — React：**

| 编号 | 问题 | 难度 | 类别 |
|------|------|------|------|
| 1 | React 中 state 和 props 的区别是什么？什么是单向数据流？ | 基础 | 概念 |
| 2 | useEffect 的依赖数组如何正确使用？常见的闭包陷阱有哪些？ | 实践 | 应用 |
| 3 | React Fiber 架构解决了什么问题？调度优先级是如何实现的？ | 原理 | 底层 |
| 4 | React.memo、useMemo、useCallback 各适用于什么场景？滥用会有什么问题？ | 实践 | 优化 |
| 5 | Concurrent Mode 下 Suspense 的工作原理是什么？与传统数据获取方式有何不同？ | 原理 | 底层 |

**技术栈问题生成覆盖规则：**

- 简历列出 N 项技术 → 至少生成 3N 个问题
- 核心技能（标注"精通"或"熟练"）→ 问题更深，侧重原理与对比
- 辅助技能（标注"了解"）→ 问题偏基础，不追问底层
- 同一生态的技术（如 Spring Boot + MyBatis + MySQL）→ 生成 2-3 个串联问题（如：Spring 事务管理在 MySQL 隔离级别下如何工作？）

### 1.3 从「项目经历」生成问题

每个项目按三大维度生成问题：

```
【项目问题分类】
  类别1 — 技术选型:   为什么选？对比过什么？各自的 trade-off？
  类别2 — 难点攻克:   最大挑战？怎么定位？怎么解决？有什么遗憾？
  类别3 — 架构决策:   如果规模扩大 N 倍？如果有新需求？如何重构？
```

**问题模板：**

**技术选型类（每个项目 2-3 个）：**
- "在这个项目中选择 [技术A] 而非 [替代方案B] 的主要考虑是什么？"
- "你对 [技术A] 和 [替代方案B] 做过对比吗？在什么场景下你会选 B 而不是 A？"
- "[技术A] 在这个项目中带来了什么好处，又引入了什么问题？"

**难点攻克类（每个项目 2-3 个）：**
- "项目中遇到的最大技术挑战是什么？你是如何定位和解决的？"
- "有没有上线后才发现的问题？当时如何应急处理，后续如何彻底修复？"
- "如果重新做这个项目，你在技术上会做哪些不同的选择？"

**架构决策类（每个项目 2-3 个）：**
- "如果这个项目的用户量突然翻 10 倍，当前架构的瓶颈会在哪里？怎么应对？"
- "如果要把这个项目从单体拆成微服务，你的拆分方案是什么？"
- "这个项目的数据模型如果新增 [某种复杂需求]，现有设计需要怎么改？"

**示例 — 从「电商秒杀系统」项目生成的问题：**

```
项目: 电商秒杀系统 (Spring Boot + Redis + Kafka + MySQL)

技术选型:
  Q1: 为什么选择 Kafka 而不是 RabbitMQ 或 RocketMQ？各自在秒杀场景下的优劣是什么？
  Q2: 为什么用 Redis 做库存预扣减而不是直接操作 MySQL？Redis 挂了你怎么办？
  Q3: 限流方案用的是令牌桶还是漏桶？为什么选这个？

难点攻克:
  Q4: 秒杀期间出现了超卖吗？是怎么发现和解决的？
  Q5: 缓存和数据库一致性你是怎么处理的？用了什么策略，有没有遇到脏数据？
  Q6: 秒杀流量峰值时系统响应变慢，你做了哪些优化？效果如何量化？

架构决策:
  Q7: 如果秒杀活动扩展到 100 万 QPS，你的架构需要怎么演进？
  Q8: 如果秒杀活动需要支持"预约+资格抽选+支付"的复杂流程，现有设计怎么改？
  Q9: 如果有恶意脚本刷接口，你的防刷方案是什么？
```

### 1.4 从「实习/工作经历」生成行为面问题

采用 STAR 方法（Situation - Task - Action - Result）设计问题：

**固定问题模板（每个实习经历 3-4 个）：**

| 类别 | 问题模板 |
|------|----------|
| 冲突处理 | "描述一次你和同事/Leader 有技术分歧的情况。你是怎么处理的？结果如何？" |
| 主导推动 | "在实习中你主导的最大改动是什么？你如何推动各方达成一致？" |
| 失败反思 | "描述一次你在工作中犯过的错误。你是怎么发现、处理、并从中学到了什么？" |
| 快速学习 | "描述一次你需要在短时间内学习一项新技术并应用的情况。你是怎么做的？" |
| 跨部门协作 | "有没有需要和其他团队（设计/产品/测试）紧密协作的经历？遇到过什么沟通问题？" |

**行为面追问策略：**
- 如果回答过于笼统 → "能举一个具体的例子吗？"
- 如果缺少量化结果 → "这个改动带来的具体效果是什么？有数据吗？"
- 如果只讲自己 → "你在这个团队中具体扮演了什么角色？"

## 二、模拟面试流程

### 2.1 启动 Phase 5

当用户输入"开始面试"或进入 Phase 5 时，展示面试类型选择：

```
┌─────────────────────────────────────────────┐
│            🎯 Phase 5 — 面试准备             │
│                                             │
│  请选择面试类型:                              │
│                                             │
│  1. 技术面 (60 min 模拟)                     │
│     - 聚焦简历所列技术栈                       │
│     - 基础 → 实践 → 原理 递进                 │
│     - 含一道简单算法/系统设计题                 │
│                                             │
│  2. 行为面 (30 min 模拟)                     │
│     - 基于实习/项目经历的 STAR 问题            │
│     - 含冲突处理、团队协作、领导力              │
│                                             │
│  3. 项目深挖 (45 min 模拟)                   │
│     - 选简历中 1-2 个核心项目深度盘问           │
│     - 技术选型 → 架构 → 难点 → 量化效果         │
│                                             │
│  4. 综合面 (90 min 模拟)                     │
│     - 技术 + 行为 + 项目深挖全覆盖             │
│     - 模拟大厂终面流程                         │
│                                             │
│  输入数字 1-4 选择，或输入 "自定义" 调整时长     │
└─────────────────────────────────────────────┘
```

### 2.2 面试执行流程

```
面试开始:
  ├── 1. 开场: 简要说明面试类型、预估时长、评分标准
  ├── 2. 提问循环 (逐个):
  │     ├── 展示当前问题
  │     ├── 等待用户输入答案
  │     ├── 对每个回答给出即时反馈:
  │     │     ├── 评分 (1-5 分)
  │     │     ├── 优点: 哪些说得好，为什么好
  │     │     ├── 改进建议: 怎么说更好 (给出具体话术示例)
  │     │     └── 追问 (如适用): 模拟面试官的深度追问
  │     └── 进入下一题
  ├── 3. 中场提示:
  │     ├── 每 5 题后显示进度 (如 "已完成 5/15 题，当前平均分 3.6/5")
  │     └── 提问节奏: 每题给 2-5 分钟思考/回答时间
  └── 4. 面试结束:
        ├── 询问: "还有想补充的吗？"
        └── 生成面试报告
```

### 2.3 即时反馈模板

每道题回答完毕后，输出以下格式的反馈：

```
───────────────────────────────────
📊 第 {N} 题反馈

评分: {X}/5 ⭐

✅ 说得好的:
  • {具体优点1 — 引用用户的话说明哪里好}
  • {具体优点2}

💡 可以这样说 (改进建议):
  • {改进点1}
  • {改进点2}

🗣️ 建议话术:
  "{一段 100-200 字的优化回答示例}"

🔍 追问:
  "{一个更深入的追问问题}"
───────────────────────────────────
```

**评分标准：**

| 分数 | 标准 |
|------|------|
| 1 | 完全无法回答，或回答严重错误 |
| 2 | 有基本概念但表达混乱，缺乏逻辑 |
| 3 | 回答正确但不够深入，缺乏细节或实例 |
| 4 | 回答完整、有逻辑、能举出实例 |
| 5 | 回答深入、有独立思考、能提出 trade-off 或对比方案 |

### 2.4 特殊场景处理

**场景1 — 用户回答"不知道"：**
- 不直接给答案
- 先给提示引导思考："你可以从 X 角度想想"
- 如果仍无法回答 → 给出标准答案，标记为"需加强"领域

**场景2 — 用户回答过长/跑题：**
- 温和提醒："你的回答覆盖了 X，但这题主要是问 Y，能否聚焦在 Y 上？"

**场景3 — 用户回答抄袭感明显（直接背诵八股文）：**
- 追问一个需要实际经验的变形："很好，那你在实际项目中有遇到过这种情况吗？"

**场景4 — 用户请求跳过当前题：**
- 允许跳过，但记录为"未回答"
- 报告中将此题列为"需复习"并附标准答案

| 总分 | 百分位 | 评价 |

| 4.5-5.0 | 90%+ | 面试表现优异，大概率通过大厂面试 |

## 四、各方向专属题库

### 4.1 后端开发

```
核心领域:
  - 操作系统: 进程/线程/协程、内存管理、文件系统、I/O 模型
  - 计算机网络: TCP/UDP/HTTP/HTTPS、DNS、负载均衡、CDN
  - 数据库: MySQL 索引/事务/锁/分库分表、Redis 数据结构/持久化/集群
  - 分布式: CAP/Base、一致性协议、分布式事务、服务治理
  - 系统设计: 短链系统、秒杀系统、消息队列、分布式 ID 生成

经典问题:
  1. "从输入 URL 到页面展示，发生了什么？" → 考察网络栈全链路理解
  2. "设计一个千万级用户的短链系统" → 考察系统设计能力
  3. "MySQL 在 RR 隔离级别下如何解决幻读？" → 考察数据库内核理解
  4. "如何设计一个分布式限流器？" → 考察分布式系统设计
  5. "微服务之间如何做服务发现和负载均衡？" → 考察微服务架构
  6. "如何保证消息队列的可靠投递？" → 考察消息系统理解
  7. "缓存穿透、缓存击穿、缓存雪崩分别是什么？如何解决？" → 考察缓存实战
  8. "Raft 协议是如何保证一致性的？" → 考察分布式一致性
```

### 4.2 前端开发

```
核心领域:
  - JavaScript 核心: 原型链、闭包、事件循环、Promise、async/await
  - 浏览器原理: 渲染流程、重排重绘、V8 垃圾回收、事件机制
  - 框架深入: React/Vue 响应式原理、虚拟 DOM、Diff 算法、编译优化
  - 性能优化: 首屏优化、懒加载、代码分割、Bundle 分析、Core Web Vitals
  - 工程化: Webpack/Vite 原理、模块化方案、CI/CD、监控体系

经典问题:
  1. "JavaScript 的事件循环机制是什么？宏任务和微任务的区别？" → 考察 JS 核心
  2. "Vue 3 的响应式系统用 Proxy 代替 defineProperty 带来了什么改进？" → 考察框架深度
  3. "从浏览器输入 URL 到页面渲染完成，前端有哪些优化点？" → 考察性能全局观
  4. "如何设计一个前端监控 SDK？" → 考察工程化能力
  5. "React Hooks 的闭包陷阱是什么？如何解决？" → 考察框架实战
  6. "Webpack 的 loader 和 plugin 有什么区别？各写一个简单示例" → 考察工程化深度
  7. "CSS 的 BFC 是什么？如何触发？有什么应用场景？" → 考察 CSS 深度
  8. "前端怎么做跨端开发？小程序、H5、App 的差异和适配方案？" → 考察多端经验
```

### 4.3 算法/机器学习

```
核心领域:
  - ML 理论: 常见算法推导、损失函数、正则化、优化器
  - 深度学习: Transformer、Attention、CNN/RNN、训练技巧
```

## 五、面试弱点 → 简历反哺机制

### 5.1 核心逻辑

面试是简历的"压力测试"。面试中暴露的问题，往往是简历中隐藏的瑕疵。Phase 5 结束后，应自动分析并生成简历修改建议。

### 5.2 反哺规则

| 面试现象 | 简历问题 | 建议操作 |
|----------|----------|----------|
| 列了某技术但无法回答基础问题 | 技术栈虚高 | 降低掌握程度（精通→熟练→了解）或移除 |
| 列了某技术但深入追问无法回答 | 技术栈"了解"不足 | 将掌握程度降为"了解"或补充学习计划 |
| 项目描述在追问中站不住脚 | 项目描述有水分或不具体 | 重写项目描述，补充量化数据和具体技术细节 |
| 面试中展示的强项未在简历体现 | 简历遗漏优势 | 在对应模块新增此技能/经历 |
| 回答好但简历无相应关键词 | 简历 ATS 匹配度低 | 在技能栈或项目描述中补充关键术语 |
| 行为面回答出色但简历未提及 | 经历挖掘不充分 | 在实习/项目经历中补充相关事迹 |
| 系统设计能力强但简历偏执行 | 简历定位与能力不匹配 | 调整简历风格，突出架构和决策能力 |
| 某方向频繁被追问且表现好 | 此方向有竞争力 | 简历中强化此方向，可作为差异化亮点 |
| 面试官明显对某段经历感兴趣 | 该经历有吸引力 | 调到更显眼位置，扩充细节 |

### 5.3 更新到飞书目标表

面试报告中标记的简历问题，应同步写入用户的飞书多维表格"目标表"进行追踪：

```
写入字段:
  - 类别: "简历优化"
  - 目标: 具体修改项（如"Redis 掌握程度从精通降为熟练"）
  - 优先级: 高/中/低（对应报告中的紧急/重要/锦上添花）
  - 截止日期: 自动设为一周后
  - 状态: 待开始
  - 来源: "Phase 5 面试反馈"
```

### 5.4 持续改进循环

```
      简历构建 (P1-2)
           │
           ▼
      项目实现 (P3-4)
           │
           ▼
      面试模拟 (P5)
           │
           ├── 表现好 → 继续下一轮 / 投递
           │
           └── 暴露问题 → 反哺简历
                            │
                            ▼
                     更新简历 + 学习补强
                            │
                            ▼
                     再次面试模拟 → 验证改进
```

## 六、实现注意事项

### 6.1 问题生成的智能化

- 问题生成应结合简历实际内容，而非无脑套模板
- 检测简历中是否有数字/百分比 → 生成"量化效果"类追问
- 检测简历中的技术组合 → 生成"技术之间协作原理"类串联题
- 检测实习公司的行业属性 → 生成行业特定的业务场景题

### 6.2 面试过程的人性化

- 不连续追问：每 3-4 个技术题后穿插 1 个轻松的行为题
- 适时鼓励：当用户表现好时给予积极反馈
- 难度自适应：如果用户连续 3 题 1-2 分 → 降低后续难度，反之提升
- 记录"亮点回答"：用户回答特别好的题目标记，报告中重点表扬

### 6.3 报告生成时机

- 面试结束立即生成
- 同时询问用户是否需要保存到飞书文档
- 报告中每个"改进建议"都附带可执行的行动计划
- 提供"导出为 Markdown"选项以便用户分享给 mentor/朋友

### 6.4 与技能的其他 Phase 的联动

- Phase 5 的面试报告可反馈到 Phase 1-2 的简历构建 → 自动生成修改建议
- Phase 3-4 的项目实现可作为"项目深挖"面试的素材来源
- 多次面试模拟后 → 生成"进步曲线"展示各轮次评分变化趋势
- 面试表现数据 → 写入飞书多维表，用于长期进度追踪
