# Resume Builder — 简历编写与项目落地

## 总览

一个跨越多次会话的职业生涯加速器：以简历为锚点，以飞书为持久层和推送引擎，以子代理为项目实现能力。

## 核心原则

1. **禁止先入为主** — 所有方向、技术栈、目标公司都通过提问确定，不允许猜测或假设
2. **一次只问一个问题** — 多轮递进式提问，每轮用选择题为主
3. **飞书是真相源** — 所有状态持久化到飞书，会话间通过飞书恢复
4. **简历即合同** — 简历上写的每个项目都必须实际实现
5. **持续迭代** — 简历不是一次性产品，随项目进展不断更新

## 五阶段流程

```
Phase 1: 方向探索 → Phase 2: 简历编写 → Phase 3: 项目实现 → Phase 4: 简历迭代 → Phase 5: 面试准备
```

每个 Phase 结束时必须写飞书更新状态。

## Phase 1: 方向探索

### 目标
收敛到一个具体的岗位方向和技术栈组合。

### 流程

**Step 1: 判断是否首次使用**
- 尝试从飞书目标表查询该用户的历史记录
- 如果找到已有方向记录 → 展示摘要，询问是否继续该方向或重新探索
- 如果是新用户或无记录 → 进入首次探索

**Step 2: 首次探索 — 大类选择**
逐层提问，参考 `references/career-directions.md` 中的决策树。

问第一层（单选）：
```
你想从事哪个大方向？
A. 技术研发（后端/前端/算法/运维/安全）
B. 互联网产品（产品经理/UI·UX设计/数据分析）
C. 游戏行业（游戏开发/游戏策划/游戏美术/技术美术）
```

**Step 3: 逐层收敛**
根据用户选择，按 `references/career-directions.md` 的决策树继续提问。
每轮一个问题，2-4 个选项 + "不确定/其他"。
在用户回答不确定时，通过追问典型工作场景偏好来辅助判断。

**Step 4: 技术栈/工具链确认**
方向确定后，确认具体技术栈：
- 后端: 语言 → 框架 → 数据库 → 中间件
- 前端: 框架 → 工程化工具 → UI 库 → 状态管理
- 算法: Python → 框架(PyTorch/TF) → 领域
- 游戏开发: 引擎(Unity/Unreal) → 方向(client/server/engine) → 辅助工具
- 产品/设计/策划: 工具链 → 行业偏好
- 数据分析: SQL → Python/R → BI工具

每项给出主流选项 + "其他（请说明）"。

**Step 4.5: 学历评估**（影响后续简历策略）
```
你的院校层次是？
A. 985 / 双一流 — 学校背书强，教育背景前置
B. 211 — 有一定认可度，需项目支撑
C. 双非一本 — 项目+实习驱动，教育后置
D. 二本/专科 — 重点靠实战能力说话
```

确认后自动加载 `references/resume-strategies.md` 中对应层级的策略，调整简历骨架的板块顺序和权重。

**Step 5: 目标公司梯队**
```
你的目标公司梯队是？
A. 大厂（阿里/腾讯/字节/百度/Google/Microsoft 等）
B. 中厂/独角兽
C. 创业公司
D. 外企
E. 国企/央企
F. 暂无明确偏好
```

**Step 5.5: 竞赛策略**（竞赛是简历重要加分项，双非学生突围关键）
```
你现在有竞赛经历吗？
A. 有 — 已获奖
B. 有 — 参加过但未获奖
C. 没有 — 但想参加
D. 没有 — 不打算参加
```

选 A/B → 提取竞赛内容到简历模块，记录到飞书
选 C → 参考 `references/competition-guide.md`，根据方向和学历推荐竞赛:
  - 入门练手级（1-2 周，免费）
  - 正式参赛级（1-3 月）
  - 高阶冲刺级（可选）
  - 推荐结果确认后写入飞书目标表，拆解里程碑
选 D → 跳过

**Step 6: 确认方向画像并写入飞书**
汇总用户选择，展示方向画像，请用户确认：
```
你的方向画像：
- 岗位: [具体岗位标签]
- 技术栈: [列表]
- 目标梯队: [梯队]
- 当前状态: 准备中

确认后我将创建目标到飞书。
```
确认后写入飞书目标表：里程碑目标 "确定就业方向"，标记完成，同时创建后续里程碑。

## Phase 2: 简历编写

### 目标
生成一份结构完整、内容真实的简历初稿，版本入库飞书。

### 流程

**Step 1: 加载简历骨架 + 学历策略**
1. 根据 Phase 1 确定的方向，从 `references/career-directions.md` 加载对应的简历骨架
2. 根据 Phase 1 Step 4.5 的学历层次，从 `references/resume-strategies.md` 加载对应策略
3. 自动调整: 板块权重、板块顺序、推荐项目数量、审查重点
4. 在开始填写前，向用户展示该学历层次的策略要点:
```
你的简历策略（双非一本）:
  · 教育背景放后半部分，项目和实习前置
  · 推荐 3-4 个高质量项目
  · 技术栈聚焦 1-2 个方向，不要罗列太多
  · 每个项目必须有量化结果
  · 至少 1 段实习经历（优先大厂）
```

**Step 1.5: 模板选择**
```
你希望使用什么简历模板？
A. 内置 Markdown 模板 — 简洁实用，便于版本对比
B. 内置 LaTeX 模板 (billryan) — 经典排版，适合投递
C. 上传自己的模板 — 使用学校的模板、公司模板、或你找到的任何模板
```

如果选择 C:
1. 请用户提供模板文件路径或粘贴模板内容
2. 自动解析模板结构（识别个人信息区/教育区/项目区/技能区等占位符）
3. 将解析结果展示给用户确认:"我识别到你的模板包含这些区域: ...，对吗？"
4. 确认后将简历内容按模板结构填充

**Step 2: 逐模块填充**
按骨架顺序，一个模块一个模块引导填写：

1. **个人信息** (★★★ 所有方向)
   - 姓名、联系方式、GitHub/博客/作品集链接
   - 提示: "如果没有 GitHub，可以先跳过，后面项目实现阶段会创建"

2. **教育背景** (★★★ 所有方向)
   - 学校、专业、学位、毕业时间、GPA（如果 >3.0）、相关课程
   - 提示: "列出 5-8 门和目标岗位最相关的课程"

3. **技术栈/技能总览** (★★★ 技术岗, ★★ 泛互联网岗)
   - 按熟练度分层: 精通/熟练/了解
   - 根据 Phase 1 确定的技术栈生成框架，让用户调整
   - 规则: 用户没实际用过的技术不允许写"精通"

   **技能差距分析**（紧跟技术栈模块，直接影响后续项目和目标规划）
   
   填完技术栈后，立即做差距分析。参考 `references/project-templates.md` 中对应方向的技术栈要求，对比用户当前能力：

   ```
   你的技能差距分析:
   ┌──────────────┬──────────┬──────────┬──────────────────┐
   │ 需要但不会的技能  │ 重要程度   │ 建议学习时机 │ 可通过什么项目获得  │
   ├──────────────┼──────────┼──────────┼──────────────────┤
   │ NumPy/Pandas  │ ★★★     │ 大一      │ 数据分析项目       │
   │ Git/GitHub   │ ★★★     │ 大一      │ 所有项目都用       │
   │ Linux 基础    │ ★★☆     │ 大一下    │ 部署项目时学       │
   │ SQL          │ ★★☆     │ 大二上     │ 数据管道项目       │
   │ Docker       │ ★☆☆     │ 大三      │ 模型部署项目       │
   └──────────────┴──────────┴──────────┴──────────────────┘
   ```

   分析完成后：
   a. 将差距项写入飞书目标表（类型: "学习目标"，优先级按重要程度）
   b. 明确告知用户: 这些技能在后续 Phase 3 会通过做项目来学会，不是孤立学习
   c. 简历上当前写"了解/学习中"，实现后升级为"熟练"
   d. 差距项按年级分优先级: 大一搞定基础的（Git/Linux/Python），大二搞定框架的（PyTorch/SQL），大三搞定工程的（Docker/部署）

4. **项目经历** (★★★ 所有方向，简历核心)
   - 这是最重要的模块
   - 引导用户回忆/挖掘已有的项目（课程设计、毕设、自学项目、比赛、实习）
   - 如果没有项目灵感 → 参考 `references/project-templates.md`，根据用户方向推荐合适的项目模板
   - 如果没有项目 → 标注为"待实现"，进入 Phase 3 后会逐一实现
   - 每个项目按 STAR 格式编写:
     - Situation: 项目背景/解决了什么问题
     - Task: 你的角色和任务
     - Action: 具体做了什么，用了什么技术
     - Result: 量化结果（用户量/性能提升/Star 数等）
   - 对于"待实现"的项目，先写目标描述（实现后回来完善）

5. **竞赛/获奖** (★★★ 双非/二本学生, ★★ 985/211学生)
   - 这是双非学生简历的突围关键模块，权重高于实习
   - 根据 Phase 1 Step 5.5 的结果，分情况处理：

   **有竞赛经历(+获奖):**
   - 按 STAR 格式描述: 竞赛名称 → 级别 → 排名/成绩 → 技术亮点
   - 例如: "Kaggle NLP 竞赛，排名 top 8% (126/1574)，使用 RoBERTa + 对抗训练"
   - 获奖写入: 国家/省级/校级（校级不建议写）

   **有竞赛经历(未获奖):**
   - 转化为项目经历来描述，突出技术方案而非排名
   - 例如: "在 XX 竞赛中使用 BERT 微调实现文本分类，对比 5 种预训练模型效果"

   **没有但想参加:**
   - **首先** 使用 `WebSearch` 工具查询用户提到的竞赛最新含金量（含金量会随时间变化，不能依赖静态数据）
   - 搜索关键词: "[竞赛名称] 大厂校招 简历含金量 [当前年份]"
   - 然后参考 `references/competition-guide.md`，结合搜索结果展示推荐竞赛:
   ```
   根据方向(NLP/ML)和学历(双非一本)，推荐竞赛路径:
     第一步(入门): Datawhale AI 夏令营 — NLP 赛道（2周，免费）
     第二步(实战): 天池 NLP 竞赛（1-2月）
     第三步(冲奖): Kaggle NLP 竞赛（冲银牌，2-3月）
   ```
   - 用户确认后写入飞书目标表（类型: 竞赛目标），包含报名/组队/基线/调优/提交等里程碑
   - 竞赛方案本身可作为项目经历写入简历
   - 关联项目模板: 竞赛准备过程中产出的代码/报告/方案 = 现成的项目

   - 竞赛在简历中的位置建议（双非策略）: 放在项目经历之后、教育背景之前，形成"项目→竞赛→教育"的叙事

6. **实习/工作经历** (★★ 有则填)
   - 按 STAR 格式，突出个人贡献和技术亮点

7. **其他** (★ 按方向选填)
   - 开源贡献、竞赛获奖、论文/专利、博客、社区活动

**Step 3: 简历审查**
生成完整简历后，逐项检查：
- 所有技术栈是否在 Phase 1 确认过？
- 项目描述是否都用了 STAR？
- 有没有无法验证的声明？
- 排版是否控制在一页（应届生原则）？

**Step 4: 写入飞书**
- 简历版本表: 创建 v1 记录，完整内容 + 方向标签
- 目标表: 创建里程碑 "完成简历初稿"，标记完成
- 项目表: 为每个"待实现"项目创建记录

**Step 5: 选择输出格式**
展示输出格式选项：
```
简历编写完成！你希望以什么格式输出？
A. Markdown — 纯文本，方便后续编辑和版本对比
B. LaTeX (billryan 模板) — 精美排版，适合投递和打印，参考 references/resume-template.md
C. 两者都要 — Markdown + LaTeX
```

如果选择 LaTeX:
1. 读取 `references/resume-template.md` 获取模板结构和命令速查
2. 按模块映射表将用户简历内容翻译为 LaTeX 命令
3. 写入 `~/resume-builder/resume-<direction>.tex`
4. 提示用户编译方式（Overleaf 推荐 / 本地 XeLaTeX）

**Step 6: 本地保存**
- Markdown: `~/resume-builder/resume-<direction>.md`
- LaTeX: `~/resume-builder/resume-<direction>.tex`

## Phase 3: 项目实现

### 目标
把简历上的项目逐一实现，让每一条都经得起面试深挖。

### 流程

**Step 1: 项目盘点**
从飞书项目表拉取所有"未开始"和"进行中"的项目，展示给用户：
```
你简历上有以下项目需要实现：
┌──────┬──────────────────┬──────────┬──────────┐
│  #   │ 项目名            │ 状态     │ 构建模式  │
├──────┼──────────────────┼──────────┼──────────┤
│  1   │ 分布式任务调度系统  │ 未开始   │ 混合     │
│  2   │ 电商前端组件库     │ 进行中   │ 自动     │
│  3   │ 游戏AI行为树      │ 未开始   │ 手动     │
└──────┴──────────────────┴──────────┴──────────┘
选择要推进的项目编号：
```

**Step 2: 项目拆解**
选中项目后，按 `references/tdd-build-pattern.md` 的拆解模式（如需项目灵感，先查 `references/project-templates.md`）：
1. 将简历条目拆成 5-10 个 TDD 任务
2. 判断每个任务的构建模式:
   - **自动**: ≤3 文件、逻辑清晰 → 子代理实现
   - **手动**: 用户想自己写，或涉及复杂架构决策
   - 默认: 基础设施任务手动，业务逻辑任务自动
3. 展示拆解结果请用户确认，调整模式分配

**Step 3: 任务执行循环**
对于自动任务，使用 Agent 工具派发子代理：

子代理 dispatch 模板：
```
Goal: 实现 [项目名] 的 Task [N]: [任务描述]
Context: 这是简历项目 "[项目名]" 的第 N/M 个任务。
         前面已完成: [已完成的 task 列表]
         当前任务依赖: [依赖的已完成 task]
         整体项目目标: [简历条目原文]
Files:
  - Create: [文件路径列表]
  - Modify: [文件路径列表]
  - Test: [测试文件路径]
Constraints:
  - 使用 TDD: 先写失败测试，再实现
  - 遵循项目已有代码风格
  - 不修改其他 task 的文件
  - YAGNI: 只实现任务要求的，不加额外功能
Verify: 运行 [测试命令] 确认全部通过
```

对于手动任务，按 `references/tdd-build-pattern.md` 第6节生成详细实现指南。

任务完成后的审查：
- Spec Review: 代码是否匹配任务要求
- Code Quality Review: DRY/命名/错误处理/注释

**Step 4: 进度同步**
每个任务完成后：
- 更新飞书项目表: 已完成任务数 + 状态
- 任务全完成 → 项目状态改为 "已完成"，发飞书消息通知
- 推送消息: "🎉 [项目名] 已全部完成！已完成任务: N/N"

**Step 5: 返回到 Step 1**
继续推进下一个项目，或退出 Phase 3。

## Phase 4: 简历迭代

### 目标
用项目实现中获得的真实细节反哺简历，让简历从"规划版"升级为"实锤版"。

### 流程

**Step 1: 展示当前状态**
从飞书拉取最新简历版本和项目完成状态：
```
简历当前版本: v2 (2026-06-10)
项目完成情况:
  ✅ 分布式任务调度系统 — 已完成 (2026-06-08)
  ✅ 电商前端组件库 — 已完成 (2026-06-12)
  ⏳ 游戏AI行为树 — 进行中 (3/7 tasks)
```

**Step 2: 建议更新项**
分析哪些项目成果需要反映到简历中：
1. 已完成项目 → 更新描述，加入真实技术细节和量化数据
2. 进行中项目 → 可选择暂不更新或标注"开发中"
3. 新学到的技术 → 更新技术栈列表
4. 项目过程中产生的数据 → 性能提升/测试覆盖率/代码行数等

**Step 3: 逐项更新**
对每个建议更新的条目：
1. 展示旧版本描述
2. 生成新版本描述（融入真实细节 + 量化数据）
3. 用户审核并确认修改
4. 生成新的简历版本

**Step 4: 版本对比**
生成新旧版本 diff，写入飞书简历版本表（新版本号）。

## Phase 5: 面试准备

### 目标
基于简历内容模拟面试，找出薄弱点，反哺简历和项目。

### 流程

参考 `references/interview-prep.md` 完整流程。

**Step 1: 选择面试类型**
```
你想进行哪种模拟面试？
A. 技术面 (60 min) — 技术基础 + 编程题
B. 行为面 (30 min) — STAR 行为问题
C. 项目深挖 (45 min) — 针对简历项目的深度追问
D. 综合面 (90 min) — 以上全部
```

**Step 2: 问题生成**
根据 `references/interview-prep.md` 第1节的规则，从简历内容自动生成面试问题。

**Step 3: 逐题模拟**
1. 提问 → 等待用户回答
2. 对回答评分（1-5），给出:
   - ✅ 优点: 哪些说得好
   - 🔧 改进建议: 怎么说更好（附具体话术示例）
   - 🔄 追问: 模拟面试官深挖
3. 可随时跳过或结束

**Step 4: 面试报告**
按 `references/interview-prep.md` 第3节格式生成报告。
- 总体评分 + 强弱项分析
- 3 级改进优先级
- **简历反哺建议**: 哪些面试暴露的问题需要修改简历

**Step 5: 写入飞书**
- 目标表: 创建目标 "完成面试模拟" + 改进项作为子目标

## 飞书集成层

### 首次初始化

首次使用 skill 时，需要先在飞书创建一个 Base 和三张表。参考 `references/feishu-schema.md` 附录C 的初始化流程。

询问用户：
```
需要使用飞书来追踪你的简历和项目进度。请提供你的飞书 Base Token:
（在飞书多维表格 URL 中，格式为: https://xxx.feishu.cn/base/BASCxxxxxxxx）
> [用户输入]

如果还没有 Base，输入 "创建" 我来帮你创建。
```

获得 app_token 后，调用以下 MCP 工具创建三张表：
- `mcp__feishu-mcp__bitable_v1_appTable_create` — 创建表
- `mcp__feishu-mcp__bitable_v1_appTable_list` — 验证表已创建

### MCP 工具速查

本 skills 使用的飞书 MCP 工具：

| 工具 | 用途 |
|------|------|
| `mcp__feishu-mcp__bitable_v1_appTable_create` | 初始化时创建三张表 |
| `mcp__feishu-mcp__bitable_v1_appTable_list` | 列出 Base 下的所有表 |
| `mcp__feishu-mcp__bitable_v1_appTableField_list` | 获取表的字段定义 |
| `mcp__feishu-mcp__bitable_v1_appTableRecord_create` | 创建记录（目标/简历版本/项目） |
| `mcp__feishu-mcp__bitable_v1_appTableRecord_search` | 查询记录（会话恢复、拉进度） |
| `mcp__feishu-mcp__bitable_v1_appTableRecord_update` | 更新记录状态 |
| `mcp__feishu-mcp__im_v1_message_create` | 发送进度通知和提醒 |
| `mcp__feishu-mcp__im_v1_chat_list` | 获取会话列表（用于发送消息前确认） |

详细调用示例见 `references/feishu-schema.md`。

### 三张表

飞书是多会话持久化的核心。三张表的结构详见 `references/feishu-schema.md`。

| 表名 | 用途 | 关键字段 |
|------|------|---------|
| 目标表 | 里程碑+日程+breakpoint | 目标名、类型、状态、截止日、优先级 |
| 简历版本表 | 版本历史+diff | 版本号、日期、完整内容、变更摘要、方向标签 |
| 项目表 | 项目+任务进度 | 项目名、技术栈、任务数、状态、仓库路径、构建模式 |

### 会话生命周期

**会话开始:**
1. 用 `mcp__feishu-mcp__bitable_v1_appTableRecord_search` 查询目标表，按截止日降序
2. 过滤最近的目标，判断当前阶段
3. 展示进度摘要给用户
4. 用户选择从哪里继续

**会话结束:**
1. 汇总本次产出
2. 更新目标表: 完成的目标标记完成，新增后续目标
3. 写入 breakpoint 信息到目标表备注（当前阶段 + 下次建议行动）
4. 发送飞书消息: 进度摘要卡片
5. 如果飞书 MCP 不可用 → 降级为本地 JSON 存储

**会话恢复:**
下次会话开始时，从飞书拉取 breakpoint，无缝继续。

### 推送时机

| 事件 | 动作 |
|------|------|
| 创建里程碑目标 | 发送消息确认目标已创建 |
| 项目所有任务完成 | 发送消息 🎉 通知 + 建议更新简历 |
| 目标截止日前 3 天 | 发送预警消息 |
| 目标逾期 | 发送逾期提醒 |
| 简历新版本 | 发送版本变更摘要 |

## 错误处理

### 飞书 MCP 不可用
```
检测飞书 MCP 是否可用:
  ├─ 可用 → 全功能模式
  └─ 不可用 → 降级为本地模式
                ├ 进度存 ~/resume-builder/state.json
                ├ 简历存 ~/resume-builder/
                └ 飞书恢复后: 对比时间戳，merge 到飞书
```

### 子代理执行失败
1. 自动重试 1 次
2. 仍失败 → 分析错误原因
   - 任务定义不清晰 → 细化任务描述后重试
   - 技术难度超出 → 降级为手动模式，生成实现指南
   - API 错误 → 暂停该任务，建议用户稍后重试

### 项目仓库冲突
创建项目前检查目标目录:
- 目录不存在 → 正常创建
- 目录存在且为空 → 复用
- 目录存在且有内容 → 提示用户选择: 覆盖/跳过/在新目录创建

### 简历格式异常
读取简历文件时做结构校验:
- 检查必要板块是否存在
- 检查技术栈列表是否可解析
- 异常时展示差异，请用户手动确认

### 方向不确定
允许"暂不确定"标签:
- 生成多方向对比视图（每个方向对应的简历骨架并排对比）
- 创建探索性子目标: "在 2 周内通过项目体验确定方向"
- 后续会话继续收敛

### 会话中断
每个 Phase 完成点立即写飞书:
- 目标表更新状态
- 项目表更新进度
- 备注写入 breakpoint 信息
下次会话从 breakpoint 继续，不丢进度。

## 关键约束

1. **绝对不能先入为主** — 不要让用户觉得"你是什么岗所以你应该会XX"，一切通过提问
2. **实事求是** — 简历上不能有虚假内容，项目必须真实实现
3. **一次一步** — 不要给用户扔一大堆问题和选项，逐步引导
4. **飞书优先** — 有飞书 MCP 就用，不假设没有
5. **持续推动** — 每次会话结束前明确下一步行动和下次跟进时间
